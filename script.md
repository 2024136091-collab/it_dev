## [슬라이드 2] 비전

실제로 겪었던 장면 하나를 먼저 말씀드리겠습니다.
저희 학교 본관 승강기 점검표를 확인하려고 국가승강기정보센터 사이트에 들어갔는데,
모바일에서는 레이아웃이 깨져서 PC방을 찾아가야 했습니다.
이 앱의 비전은 **"승강기 안전 정보를 누구나 손 안에서 빠르게 확인할 수 있는 세상"** 입니다.
우리나라에는 현재 80만 대 이상의 승강기가 운행되고 있지만,
정보를 조회할 수 있는 서비스는 PC 전용 웹에만 있어서 모바일에서는 이용이 불편합니다.
이 문제를 Flutter와 공공 API로 해결하고자 했습니다.

---

## [슬라이드 3] 왜 이 주제를 골랐나

방금 말씀드린 것처럼, 승강기 정보가 여러 사이트에 분산되어 있고, 기존 서비스는 모바일을 지원하지 않습니다.
건물 관리자 입장에서 생각해보면 더 분명해집니다.
입주민이 "승강기 언제 점검했냐"고 물으면 관리자는 PC를 켜고 사이트에 로그인해서 확인해야 하는데,
현장에서 스마트폰으로 바로 확인할 수 있다면 민원 대응 시간이 훨씬 줄어듭니다.
또 건물명이나 호기 번호로 빠르게 검색하는 기능도 기존 서비스에는 없었습니다.
Flutter 앱 하나로 이 세 가지 문제를 한 번에 해결할 수 있다고 생각해서 선택했습니다.

## [슬라이드 5] 활용 방안

이 앱은 건물 관리자, 입주민, 안전 점검원, 민원 담당자 등 다양한 분들이 활용할 수 있습니다.
예를 들어 관리자는 담당 건물의 점검 이력을 현장에서 바로 확인하고,
입주민은 아파트 승강기 운행 상태를 스마트폰으로 조회할 수 있습니다.

## [슬라이드 6] 핵심 기능

핵심 기능은 총 여덟 가지입니다.
건물명·주소·번호 검색, 스마트 자연어 검색, 지역·유형·종류 필터,
건물별 그룹핑, 점검 이력 조회, 위치 지도 조회, 즐겨찾기입니다.
특히 **스마트 자연어 검색**은 "오래된 승강기 찾아줘"처럼 문장으로 입력하면
키워드를 분석해서 결과를 반환하는 기능입니다.

---

## [슬라이드 7–8] 아키텍처 및 프로젝트 구조

아키텍처는 **4레이어 단방향 의존 구조**를 채택했습니다.
Presentation → Application → Domain → Data 방향으로만 의존합니다.
이렇게 하면 나중에 실제 API로 교체할 때 Data 레이어만 수정하면 되고,
나머지 레이어는 변경이 필요 없습니다.
화면은 총 5개이고, 상태 관리는 Riverpod의 StateNotifier를 사용했습니다.

---

## [슬라이드 9–10] API 및 데이터 모델

국가승강기정보센터의 승강기 번호는 `XXXX-XXX` 형식입니다.
앞자리 숫자로 지역을 판별할 수 있어서, 별도 API 호출 없이
ID만으로 서울·경기·인천 등 권역을 자동으로 구분합니다.
현재는 8개 건물, 16개 승강기 더미 데이터로 개발하고 있습니다.

## [슬라이드 11] Riverpod 상태 관리

상태 관리는 Riverpod의 `StateNotifier`와 `AsyncValue`를 사용했습니다.
`AsyncValue` 하나로 로딩·에러·데이터 세 가지 상태를 처리할 수 있어서
코드가 훨씬 간결해졌습니다.

---

## [슬라이드 12] 스마트 자연어 검색

스마트 검색은 실제 AI가 아닌 **규칙 기반 키워드 매칭**입니다.
"오래된"이나 "노후"가 포함되면 설치연도 오름차순으로 정렬하고,
"장애인"이나 "휠체어"가 포함되면 장애인용 승강기만 필터링합니다.
키워드가 매칭되지 않으면 일반 검색으로 fallback됩니다.
UI에는 보라색 테마, 예시 질문 칩, 점 애니메이션을 넣어서 AI처럼 보이도록 했습니다.

## [슬라이드 13] 구현 화면

화면은 홈, 스마트 검색, 검색 결과, 상세 정보, 즐겨찾기 다섯 개입니다.
검색 결과 화면에서는 지역·유형·종류 3단 필터와 건물별 그룹핑을 제공하고,

---

> **예상 질문 대비**
> - Q: 실제 AI가 아닌데 AI라고 표시해도 되나요?
>   A: UI의 "AI" 배지는 기능 명칭이고, 슬라이드에서 "규칙 기반"임을 명시했습니다.
> - Q: 실제 API 연동은 언제 되나요?
>   A: 13주차에 국가승강기정보센터 API 키를 발급받아 ApiRepository를 교체할 예정입니다.
> - Q: flutter_map과 Google Maps 차이는?
>   A: Google Maps는 API 키와 과금이 필요하고 Web 지원이 실험적입니다. flutter_map은 키 없이 무료로 Web에서 완전히 동작합니다.
> - Q: 왜 4레이어 아키텍처를 선택했나요?
>   A: ADR-001에 기록했듯이, 더미 데이터에서 실제 API로 교체 시 Data 레이어만 수정하면 나머지가 불변이라 유지보수가 쉽습니다.
> - Q: 왜 Provider나 Bloc이 아니라 Riverpod을 선택했나요?
>   A: ADR-002에 비교표로 정리했는데, Provider는 BuildContext에 의존해서 전역 오염이 생기고 Bloc은 학습 난이도가 높습니다. Riverpod은 BuildContext 없이도 동작하고 AsyncValue로 로딩·에러·데이터 3가지 상태를 한 타입으로 처리할 수 있어서 선택했습니다.
> - Q: 왜 처음부터 실제 API로 개발하지 않았나요?
>   A: ADR-003에 기록했듯이, 국가승강기정보센터 API는 인증키 발급과 CORS 설정이 필요해서 초기 개발 속도가 느려집니다. 더미 데이터를 실제 API 번호 체계(XXXX-XXX)에 맞춰 만들어두면 ApiRepository만 교체해서 바로 전환할 수 있습니다.
